“Naming Parametric Abstractions”

Programming is just “Naming Parametric Abstractions”. We have now seen two kinds of (named and parametric) abstractions:

Methods

Methods allow us to abstract away the specific XXXXXXXX of a method body: we can simply call the method again instead of typing again the full body. The method has both a name and named parameters.

Subtyping

Subtyping allows us to abstract types into categories: when mentioning Direction as a XXXXX we mean any of the values XXXXXXX Direction. The type has both a name and named subtypes. For example, for the type Direction, the subtypes are North, East, South and West. Nort declares to be a subtype of Direction by naming direction: North:Direction{East}

We will see other forms of abstraction later on.

Syntactic sugar

Any feature aiming to skip or simplify code is called XXXXXXXX. When we write North:Direction{.turn->East} instead of the more verbose XXXXXXXXXXXX we are relying on syntactic sugar. The same as when we write North:{East}. Here the sugar also XXXX .turn-> and the ending ;.

We have seen how we wrote:

Rotation:{ #(d: Direction): Direction; }
Rotate90: Rotation{dir-> dir.turn }

It is very common to have to XXXXX a single method with a single parameter by directly calling methods on that parameter. There is a convenient sugar for this case! :: We can write just:

Rotate90: Rotation{::turn }

This means that we can write:

Rotate180: Rotation{::turn::turnXXXXXX}

to rotate twice and even

Rotate0: Rotation{::}

to not rotate. That is, Rotate0: Rotation{::} is equivalent to Rotate0: Rotation{#(dir)->dir}.

Using {::} is very common in Fearless. Another useful form of sugar allows to omit XXXX brackets when calling methods with a single parameter, or when defining methods with inferred types. For example we can write

Rotate90 + Rotate90  //shorter
Rotate90 +(Rotate90) //with explicit brackets

Rotate90: Rotation{# dir -> dir.turn } //shorter
Rotate90: Rotation{XXXX-> dir.turn } //with explicit brackets
